第一章

Codex 是什么

重新定义 AI 的角色：从"回答者"到"执行者"

Codex 是一个自动化执行系统，由三大核心组成：

🧠
Skill
能力
  • 写文章
  • 剪视频
  • 生成封面
  • 抓取网页
  • 处理文本
🔧
MCP
工具
  • 调用 FFmpeg
  • 调用 Firecrawl
  • 调用 Notion API
  • 调用浏览器
  • 调用本地命令
🎯
scripts
调度
  • 顺序控制
  • 条件判断
  • 重试机制
  • 结果检查
  • 产出管理
💡 核心理念

Codex 不是一个"回答问题的 AI"，而是一个能执行任务、能调度工具、能自动化生产内容的系统。它让 AI 从"回答"变成"行动"。

Codex 的六大价值：

  • 自动执行 — 无需人工干预完成复杂任务
  • 自动判断 — 根据结果决定下一步操作
  • 自动修复 — 遇到错误自动尝试修复
  • 自动重试 — 失败后自动重新执行
  • 自动清理 — 完成后的临时文件清理
  • 自动产出 — 直接交付最终成品

第四章

Skill（能力层）

能力专家：只做单项能力，不做流程控制

🔁 Skill 的核心流程

① 接收 scripts 的任务描述
② 调用 MCP
③ 执行能力
④ 返回结构化数据（JSON）

Skill 的特点：

  • 不做流程 — 不决定什么时候执行
  • 不做判断 — 不评估结果好坏
  • 不做调度 — 不编排执行顺序
  • 只做"单项能力" — 专注一件事做到极致
  • 必须返回结构化数据（JSON） — 确保下游可消费
Skill 就像一个高级技工：接到任务 → 拿工具 → 干活 → 交出结果。但不问"为什么做"也不管"下一步是什么"。
第五章

MCP（工具层）

工具接口：Skill 与外部世界之间的桥梁

🔁 MCP 的核心流程

① 接收 Skill 的工具调用
② 执行命令或 API
③ 返回结构化数据（JSON）

MCP 的核心价值：

  • 解耦 — Skill 不需要知道工具的具体实现
  • 标准化 — 统一的接口规范，输入输出都是 JSON
  • 可扩展 — 新增工具只需实现 MCP 接口
  • 闭环 — Skill → MCP → 工具 → MCP → Skill 形成完整闭环
第六章

scripts（调度层）

指挥官：整个系统的"大脑"

scripts 负责的 8 项关键决策：

1
读取素材
从 assets 目录获取待处理的原始内容
2
读取参数
从 config 目录读取自动化参数配置
3
选择 Skill
根据任务类型选择合适的 Skill 模块
4
下达任务
将任务描述传递给 Skill 执行
5
检查结果
验证 Skill 返回的 JSON 是否符合要求
6
重试/修正
结果不符合要求时，自动重试或调用其他 Skill
7
写入 output
将最终成品写入 output 目录
8
清理 runtime
删除临时文件，保持环境整洁

第九章

深度解析

完整讲解 Codex 的设计哲学

🎓 设计哲学之一：关注点分离

Codex 将能力、工具、调度彻底分离。 每一层只做好自己的事，不越界。这种设计使得任意一层都可以独立替换、独立升级、独立测试。 更换一个 Skill 不影响 scripts；升级一个 MCP 不影响 Skill。

🏗️ 设计哲学之二：结构化数据驱动

所有层间通信都使用JSON。这不是一个技术选择，而是一个契约。 JSON 确保了任何 Skill 的输出都可以被任何 scripts 消费——无论 Skill 内部用什么语言实现。

🔄 设计哲学之三：自愈能力

传统自动化是"如果 A 则 B，否则失败"。Codex 是"如果 A 则 B，否则再试一次"。 这种自愈能力是 Codex 区别于普通脚本的核心差异——它能自动执行、自动判断、自动修复、自动重试。

📊 设计哲学之四：全链路可观测

从 assets 到 output 的每一步都是可追踪的。runtime 保存临时文件，scripts 保留执行日志。 你可以看到每一步的输入、输出和中间状态——这让调试变得简单，也让审计成为可能。
