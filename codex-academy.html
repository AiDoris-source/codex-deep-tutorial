第1章
Codex 是什么
重新定义 AI 的角色：从"回答者"到"执行者"
🧠 理解🏗️ 独立完成⚡ 低负荷 · 15min

Codex 是一个自动化执行系统，由三大核心组成：Skill（能力）、MCP（工具）、scripts（调度）。

它不是一个"回答问题的 AI"，而是一个能执行任务、能调度工具、能自动化生产内容的系统。

Codex 的六大价值

  • 自动执行 — 无需人工干预完成复杂任务
  • 自动判断 — 根据结果决定下一步操作
  • 自动修复 — 遇到错误自动尝试修复
  • 自动重试 — 失败后自动重新执行
  • 自动清理 — 完成后的临时文件清理
  • 自动产出 — 直接交付最终成品

第2章
Codex 的三层架构
能力层 · 工具层 · 调度层 —— 三位一体
🧠 理解🏗️ 独立完成⚡ 中负荷 · 20min

Codex 底层由三层组成：

🎯 scripts（调度层 · 指挥官）

决定什么时候做、用什么做、怎么做。负责顺序、条件、循环、重试、检查、产出。

🧠 Skill（能力层 · 能力专家）

只做单项能力 — 写文章、剪视频、生成封面、抓取网页。不做流程控制，必须返回结构化数据（JSON）。

🔧 MCP（工具层 · 工具接口）

调用 FFmpeg、Firecrawl、Notion API、浏览器、本地命令。是 Skill 与外部工具之间的桥梁。

数据流闭环：scripts → Skill → MCP → 外部工具 → MCP → Skill（JSON）→ scripts → output

第3章
Workspace 全结构解析
Codex 的 Workspace 是一个完整的自动化工程
🧠 理解🏗️ 独立完成⚡ 低负荷 · 15min

Codex 的 Workspace 由 8 个目录组成：

目录职责
assets要处理什么
config怎么处理
templates做成什么样
scripts什么时候做、怎么做

runtime临时文件区

数据流路线：assets + config + templates → scripts → skill → mcp → output

第4章
Skill（能力层）
能力专家：只做单项能力，不做流程控制
🧠 理解🏗️ 独立完成⚡ 中负荷 · 18min

Skill 的核心流程

  1. 接收 scripts 的任务描述
  2. 调用 MCP
  3. 执行能力
  4. 返回结构化数据（JSON）

Skill 的特点

  • 不做流程 — 不决定什么时候执行
  • 不做判断 — 不评估结果好坏
  • 不做调度 — 不编排执行顺序
  • 只做"单项能力" — 专注一件事做到极致
  • 必须返回结构化数据 — 确保下游可消费
Skill 就像一个高级技工：接到任务 → 拿工具 → 干活 → 交出结果。但不问"为什么做"也不管"下一步是什么"。

🔍 纠错演练

🐛 下面这段描述哪里有问题？
"Skill 是整个 Codex 系统的大脑，它负责决定什么时候执行任务、检查任务结果是否合格、并在失败时自动重试。"
❌ 错误！Skill 不做流程控制、不做判断、不做调度。scripts 才是大脑，负责决策和调度。Skill 只做单项能力执行。
第5章
MCP（工具层）
工具接口：Skill 与外部世界之间的桥梁
🧠 理解🏗️ 独立完成⚡ 低负荷 · 12min

MCP 的核心流程

  1. 接收 Skill 的工具调用
  2. 执行命令或 API
  3. 返回结构化数据（JSON）

MCP 的核心价值

  • 解耦 — Skill 不需要知道工具的具体实现
  • 标准化 — 统一的接口规范，输入输出都是 JSON
  • 可扩展 — 新增工具只需实现 MCP 接口
  • 闭环 — Skill → MCP → 工具 → MCP → Skill 形成完整闭环
第6章
scripts（调度层）
指挥官：整个系统的"大脑"
🧠 分析🏗️ 独立完成⚡ 中负荷 · 20min

scripts 负责的 8 项关键决策：

  1. 读取素材 — 从 assets 目录获取原始内容
  2. 读取参数 — 从 config 读取自动化配置
  3. 选择 Skill — 根据任务类型选择合适的 Skill
  4. 下达任务 — 将任务描述传递给 Skill
  5. 检查结果 — 验证 Skill 返回的 JSON 是否符合要求
  6. 重试/修正 — 结果不符合要求时自动重试
  7. 写入 output — 将最终成品写入 output 目录
  8. 清理 runtime — 删除临时文件
🐛 纠错题：如果 Skill 返回了不合格的 JSON，scripts 应该直接放弃并把半成品写入 output。这个说法对吗？
❌ 不对！scripts 应该先重试或调用其他 Skill 修正，而不是放弃或写入半成品。这正是 Codex 自愈能力的体现。
第7章
Codex 自动化 8 步流程
完整的自动化闭环 —— 从素材到成品
🧠 分析🏗️ 独立完成⚡ 中负荷 · 18min

这是 Codex 的核心运行机制：

  1. 📥 读取素材（assets）
  2. ⚙️ 读取参数（config）
  3. 🧠 选择 Skill
  4. 📝 下达任务描述
  5. 🔧 Skill + MCP 执行
  6. 📊 返回 JSON
  7. ✅ 检查结果
  8. 🔄 重试/修正（如需要）
  9. 📤 写入 output
  10. 🧹 清理 runtime
这是一个完整的自动化闭环。从素材输入到成品输出，每一步都由 Codex 自动完成。

第9章
深度解析
完整讲解 Codex 的设计哲学
🧠 评价🏗️ 独立完成⚡ 高负荷 · 20min

🎓 设计哲学一：关注点分离

Codex 将能力、工具、调度彻底分离。每一层只做好自己的事，不越界。任意一层都可以独立替换、独立升级、独立测试。

🏗️ 设计哲学二：结构化数据驱动

所有层间通信都使用 JSON。任何 Skill 的输出都可以被任何 scripts 消费——无论 Skill 内部用什么语言实现。

🔄 设计哲学三：自愈能力

"如果 A 则 B，否则再试一次"。这种自愈能力是 Codex 区别于普通脚本的核心差异。

📊 设计哲学四：全链路可观测

从 assets 到 output 的每一步都是可追踪的。runtime 保存临时文件，scripts 保留执行日志。

🔄 迁移任务

把三层架构用到你自己的工作中

想一个你日常重复做的工作任务（比如每周整理报表、每天回复邮件模板、定期更新数据），用 Codex 的三层模型重新设计它：谁是指挥官（scripts）？什么是能力模块（Skill）？需要什么工具（MCP）？

【真实案例】

一家小团队每周一开运营会。会后负责人需要花 40 分钟：从聊天记录里找待办事项、按负责人和截止时间整理、写会议摘要。Codex 可以直接在项目文件夹中读取会议记录，生成 行动清单.md 和 会议摘要.md。

  1. 为什么"先阅读，不要修改"是一个重要指令？

接入 OpenAI API：生成摘要与平台草稿
🧠 应用🏗️ 独立⚡ 高·50min

【目标】

  • 理解 API 请求和响应
  • 在服务器端完成一次 OpenAI API 调用
  • 正确保护 API Key、费用和输入数据

核心安全原则

  • API Key 不进入前端代码、截图和 Git
  • 请求失败时给出普通用户能理解的提示
  • 有加载状态，避免重复点击
  • 限制输入长度
  • 不上传客户秘密或不必要的个人信息
  • 设置使用额度与费用提醒
  • 结果只保存为草稿
🐛 找错题
const API_KEY = "sk-abc123..." // 写在网页 JS 文件里
❌ 致命错误！API Key 绝对不能写在前端代码里。任何人打开浏览器开发者工具就能看到你的 Key。正确做法：Key 放在服务器端环境变量中，前端通过自己的服务器中转调用。
